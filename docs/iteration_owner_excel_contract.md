# 迭代模块需求负责人 Excel 配置规范

每个迭代上传一份需求负责人 Excel。这个文件只描述本迭代的模块需求、前端负责人姓名、后端负责人姓名和需求优先级；人员角色、手机号、企微 UserID 等通讯录信息统一维护在 GitHub 配置中。

内置 AI 读取文档变更摘要和本 Excel 后，负责识别命中的模块需求、前端负责人和后端负责人；脚本再按姓名去 GitHub 人员配置里补齐角色与企微 @ 信息，并按通知策略追加测试负责人，最后发送企业微信通知。

模板文件：[iteration_module_owners_template.xlsx](./iteration_module_owners_template.xlsx)

完整 Agent 操作手册见：[ai_agent_operation_manual.md](./ai_agent_operation_manual.md)

需求文档版本归档规则见：[iteration_requirement_versioning.md](./iteration_requirement_versioning.md)

## 推荐工作流

1. 迭代开始前，项目负责人为本迭代建立目录，例如 `docs/iterations/S1/00_initial/`。
2. 在 `00_initial/` 中保存最初需求和本迭代 Excel。
3. 每次技术文档发生变更后，自动摘要脚本生成 `docs/changes.json`。
4. 每次需求变动新增版本目录，例如 `docs/iterations/S1/v1/`，不要覆盖旧文件。
5. 内置 AI 读取 `docs/changes.json`、当前迭代最新版本目录和本迭代 Excel。
6. AI 根据文档变更内容匹配 Excel 中的 `模块需求`。
7. AI 输出命中的 `模块需求`、`前端负责人`、`后端负责人` 和 `需求优先级`。
8. 脚本按姓名查询 GitHub 人员配置，补齐 `角色`、`手机号`、`企微UserID`。
9. 通知脚本按角色分组，并用手机号或企微 UserID 发送原生 @。

## Sheet：迭代需求负责人

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| 模块需求 | 必填 | 一句话或一段话描述模块、需求、接口、页面或字段变化范围。AI 用它和文档变更摘要做语义匹配。 |
| 前端负责人 | 按需填写 | 只填前端负责人姓名；多个姓名用顿号、逗号或分号分隔，例如 `晋鹏、张三`。 |
| 后端负责人 | 按需填写 | 只填后端负责人姓名；多个姓名用顿号、逗号或分号分隔，例如 `聂巧宇、李四`。 |
| 需求优先级 | 建议填 | 推荐 `P0`、`P1`、`P2`、`P3`。多个需求命中时，优先级高的排前面。 |

## 填写建议

- `模块需求` 尽量写清楚业务模块、接口路径、页面名、字段名或核心业务词，例如 `订单中心：调整支付回调字段，补充异常状态处理说明`。
- `前端负责人`、`后端负责人` 只写姓名，不写手机号、不写角色、不写企微账号。
- 一个需求可以只填前端或只填后端；另一列留空即可。
- 姓名必须和 GitHub 人员配置中的姓名一致，否则脚本无法补齐手机号和角色。
- 每个模块需求一行；一个人负责多个需求时，在多行里重复写这个人的姓名即可。
- 测试是否“每次都通知”不再由 Excel 控制，而是由 GitHub 人员/通知策略配置控制。

## AI 匹配规则

AI 应按以下优先级判断负责人：

1. 读取 Excel 的 `模块需求`、`前端负责人`、`后端负责人`、`需求优先级`。
2. 根据文档变更摘要、文件路径、影响模块、接口名、字段名，匹配最相关的 `模块需求`。
3. 命中多个需求时，按 `需求优先级` 排序：`P0 > P1 > P2 > P3 > 空值`。
4. 将 `前端负责人` 和 `后端负责人` 分别拆分成姓名数组，分隔符支持 `、`、`,`、`，`、`;`、`；`。
5. 输出命中的需求和姓名列表，不在 AI 输出里编造手机号或角色。
6. 脚本再按姓名查询 GitHub 人员配置，补齐角色和企微 @ 信息。
7. 如果 GitHub 通知策略配置了 `always_include_roles: ["测试"]`，脚本会额外追加测试负责人。

## AI 输出给脚本的 JSON

```json
{
  "version": 10,
  "changes": [
    {
      "file": "docs/demo.md",
      "matched_requirements": [
        {
          "module_requirement": "Demo 健康检查：新增 /demo/status 接口，补充 status 响应字段说明。",
          "priority": "P1",
          "frontend_owner_names": ["晋鹏"],
          "backend_owner_names": ["聂巧宇"]
        }
      ]
    }
  ]
}
```

## 通知展示

脚本根据 GitHub 人员配置补齐后，上方 markdown 通知展示角色分组：

```text
相关负责人：
测试：沈阳
前端：晋鹏
后端：聂巧宇
```

下方 text 消息只发送统一提醒：

```text
请相关负责人关注技术文档更新 v10，详情见上方通知。
```

真正的企微蓝色 @ 由 `mentioned_mobile_list` 或 `mentioned_list` 生成，避免正文里出现重复的黑色 @ 文案。

## GitHub 人员配置职责

GitHub 配置负责维护稳定人员主数据，示意如下：

```json
{
  "people": [
    {"name": "沈阳", "role": "测试", "mobile": "***********", "user_id": ""},
    {"name": "晋鹏", "role": "前端", "mobile": "***********", "user_id": ""},
    {"name": "聂巧宇", "role": "后端", "mobile": "***********", "user_id": ""}
  ],
  "notification_policy": {
    "always_include_roles": ["测试"]
  }
}
```
